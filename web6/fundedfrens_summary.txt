OVERVIEW
--------
FundedFrens is a crypto prop firm platform for Solana meme coin traders.
Users pay a one-time fee to take a trading challenge. If they pass, they
get access to funded SOL capital and keep up to 90% of profits.

Brand direction: Premium, clean, modern, trustworthy.
Design inspiration: Stripe, Vercel, Linear, Coinbase.
Avoids flashy "meme coin" aesthetics.

API ROUTERS

CORE SERVICES
-------------
challenge.py
  - CRUD for challenge_plans table
  - Activating user_challenge records on payment confirmation
  - Snapshots all rules at purchase time (so plan rule changes don't affect active challenges)

payment.py
  - Creates orders with live SOL price conversion (USD → SOL)
  - Polls Solana mainnet RPC for incoming payments (30-min expiry window)
  - Verifies exact SOL amount (2-lamport tolerance for floating point)
  - On confirmation: updates order/payment, activates challenge, processes referral commission

solana_rpc.py
  - Direct Solana mainnet JSON-RPC calls (no API key for basic use)
  - getSignaturesForAddress — scans recent wallet transactions
  - getTransaction — parses SOL transfer amounts by tracking balance deltas
  - Note: Public RPC used; production should use Helius or QuickNode

sol_price.py
  - Fetches live SOL/USD price to calculate exact SOL required per plan
  - Results cached in sol_price_cache table

rule_engine.py
  - Evaluates trading rules per user challenge:
      PORTFOLIO_DRAWDOWN — static hard breach; fails challenge immediately
      DAILY_DRAWDOWN     — daily hard breach; fails challenge immediately
      PROFIT_TARGET      — marks challenge as passed
      POSITION_SIZE      — exceeding limit freezes challenge
      OPEN_POSITIONS     — exceeding limit freezes challenge
  - Phase 3 will call this after every trade recorded via Telegram bot

referral.py
  - Unique referral code generated per user on registration
  - Commission (% of plan price) credited on confirmed payments
  - Earnings: pending → available → paid
  - Payout requests to user Solana wallet

audit.py
  - Logs every significant action with user_id, action type, and details
  - Events include: payment_confirmed, challenge_activated, challenge_failed,
    challenge_frozen, account_freeze, account_reactivation, etc.

MIGRATION 002 — ROW LEVEL SECURITY
------------------------------------
RLS enabled on ALL tables.

is_admin() function:
  Returns TRUE if current auth.uid() maps to a profile with role='admin'

prevent_role_self_escalation trigger:
  Prevents authenticated users from changing their own role column.
  Service role (FastAPI backend) bypasses this.

Policies summary:
  - Users: can only read/write their own rows
  - Admins: bypass all restrictions via is_admin()
  - Service role: full access (used by FastAPI backend)
  - Challenge plans: public read for active ones
  - Audit logs: insert only via service role; read own or admin

PAYMENT FLOW (END-TO-END)
================================================================

1. User selects a plan → clicks "Buy Challenge"
2. Backend creates an order, fetches live SOL/USD price, calculates exact SOL amount
3. A 30-minute payment window is created
4. User sees treasury wallet address + QR code on /pay/[orderId]
5. Frontend polls /api/v1/orders/{id}/payment-status every few seconds
6. Backend scans treasury wallet's recent Solana transactions via mainnet RPC
7. When a matching payment is found (within 2-lamport tolerance), it's confirmed
8. Order → "activated", user_challenge record created, referral commission processed
9. User gains access to their funded challenge account

NOTABLE TECHNICAL DECISIONS
================================================================

1. NO STRIPE/PAYPAL — Payments are 100% native SOL on-chain transfers.
   Users send SOL directly to a treasury wallet. Backend verifies via Solana RPC.

2. NO HARDCODED PLANS — Challenge plans are entirely database-driven.
   Admins can create, edit, price, and activate/deactivate plans without code changes.

3. RULE SNAPSHOTS — When a user purchases a challenge, all rules are snapshotted
   into challenge_rules. Plan rule changes don't affect active challenges.

4. PUBLIC SOLANA RPC — Currently uses api.mainnet-beta.solana.com (no API key).
   Production should upgrade to Helius or QuickNode for reliability and rate limits.

5. FOREIGN KEY CHAIN — Orders always reference challenge_plan_id.
   Architecture: challenge_plans → orders → user_challenges → payments

6. RLS EVERYWHERE — Every table has Row Level Security enabled.
   FastAPI uses service role key to bypass RLS. Frontend uses anon key (RLS enforced).
